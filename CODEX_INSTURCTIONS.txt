\section{Project Overview}

\subsection{Project Name}

The proposed project name is:

\begin{center}
\textbf{PhaseGuard}
\end{center}

The name suggests that the system protects the useful acoustic signal by analyzing phase structures. It is short, understandable, and connected to the mathematical idea behind the project.

\subsection{One-Sentence Description}

PhaseGuard is an unsupervised signal-analysis tool for underwater audio recordings. It detects structured periodic interference using Fourier phase analysis and roots-of-unity cancellation patterns, then attenuates the detected components and reconstructs a cleaner version of the audio.

\subsection{The Problem}

Underwater audio recordings are often very noisy. Hydrophones can record many different sources at the same time, such as:

\begin{itemize}[leftmargin=*]
    \item ship engines,
    \item propellers,
    \item sonar signals,
    \item waves,
    \item underwater machinery,
    \item marine animal sounds,
    \item weak biological activity,
    \item random environmental noise.
\end{itemize}

The main problem is that strong periodic mechanical noise can dominate the recording. This makes weaker biological or environmental signals harder to inspect.

For example, a ship engine can create repeating frequency patterns that hide quieter sounds. A human reviewer or a downstream analysis system may then have difficulty understanding what is present in the recording.

\subsection{The Proposed Solution}

PhaseGuard does not try to directly classify species or recognize specific animals. Instead, it focuses on detecting and suppressing \textbf{structured periodic interference}.

The full conceptual pipeline is:

\[
\text{Raw audio}
\rightarrow
\text{FFT}
\rightarrow
\text{Frequency and phase components}
\rightarrow
\text{Roots-of-unity phase mapping}
\rightarrow
\text{Near-vanishing sum detection}
\rightarrow
\text{Interference attenuation}
\rightarrow
\text{Inverse FFT}
\rightarrow
\text{Cleaner audio}
\]

This makes the project mathematically interesting, but still useful in a real-world scenario.

\subsection{Main Scientific Claim}

The team should use the following claim:

\begin{quote}
PhaseGuard detects structured phase relationships in underwater audio and uses them to suppress periodic interference, making weak acoustic signals easier to inspect.
\end{quote}

The team should avoid saying that the system directly detects fish, whales, or specific species. That would be too strong and not realistic for the hackathon prototype.

\subsection{What the Project Is Not}

It is important to be honest about what the project does and does not do.

PhaseGuard is not:

\begin{itemize}[leftmargin=*]
    \item a fish detector,
    \item a whale classifier,
    \item a complete marine biology monitoring system,
    \item a perfect audio source separator,
    \item an AI model trained on labeled sound classes.
\end{itemize}

PhaseGuard is:

\begin{itemize}[leftmargin=*]
    \item an unsupervised signal-analysis prototype,
    \item a mathematical method for detecting structured phase patterns,
    \item a preprocessing tool for underwater acoustic recordings,
    \item a system that can make weak signals easier to inspect after reducing structured noise.
\end{itemize}

\section{Why This Idea Works for the Hackathon}

\subsection{Originality}

Many hackathon projects use standard artificial intelligence ideas such as image classification, text classification, or object detection. PhaseGuard is different because it uses a mathematical signal-processing idea.

The project combines:

\begin{itemize}[leftmargin=*]
    \item Fourier analysis,
    \item phase components,
    \item roots of unity,
    \item near-vanishing sums,
    \item audio reconstruction.
\end{itemize}

This gives the project a unique technical identity.

\subsection{No Labeled Data Needed}

A very strong advantage is that the core method does not need labeled data.

Many machine-learning systems require examples such as:

\begin{itemize}[leftmargin=*]
    \item this sound is a whale,
    \item this sound is a ship,
    \item this sound is a dolphin,
    \item this sound is background noise.
\end{itemize}

Creating such labels is expensive and slow. In marine environments, it can also require expert knowledge.

PhaseGuard avoids this issue because the main algorithm is unsupervised. It searches for mathematical structure directly inside the signal.

\subsection{Explainability}

The method is explainable because the detected patterns can be visualized.

The system can show:

\begin{itemize}[leftmargin=*]
    \item the original waveform,
    \item the frequency spectrum,
    \item selected phase components,
    \item points on the unit circle,
    \item groups of vectors whose sum is close to zero,
    \item the reconstructed cleaned audio.
\end{itemize}

This is useful for the hackathon because judges can see what the system is doing. It is not just a black-box model returning a label.

\subsection{Good Demo Potential}

A good hackathon project needs a demo that is easy to understand. PhaseGuard can provide a strong demo because it has both visual and audio output.

The demo can show:

\begin{enumerate}[leftmargin=*]
    \item the original noisy audio,
    \item the frequency spectrum,
    \item detected structured interference,
    \item the unit-circle explanation,
    \item the cleaned reconstructed audio,
    \item before-and-after audio comparison.
\end{enumerate}

This makes the project more memorable.

\section{Technical Concept}

\subsection{Fourier Transform}

The Fourier transform decomposes an audio signal into frequency components.

Instead of only looking at the sound over time, we can analyze:

\begin{itemize}[leftmargin=*]
    \item which frequencies are present,
    \item how strong each frequency is,
    \item what phase each frequency has.
\end{itemize}

Periodic mechanical sounds, such as engines and propellers, often create strong repeating frequency patterns. These patterns are visible in the Fourier domain.

\subsection{Phase Components}

Each frequency component has a phase. A phase can be represented as a point on the unit circle:

\[
e^{i\theta}
\]

where \(\theta\) is the phase angle.

When phases are discretized, they can be approximated by roots of unity:

\[
\zeta_N^k = e^{2\pi i k/N}
\]

where:

\begin{itemize}[leftmargin=*]
    \item \(N\) is the number of phase divisions,
    \item \(k\) is the index of the root,
    \item \(\zeta_N^k\) is a point on the unit circle.
\end{itemize}

\subsection{Roots of Unity}

Roots of unity are evenly spaced points on the unit circle.

For example, for \(N = 3\), the roots are:

\[
1,\quad e^{2\pi i/3},\quad e^{4\pi i/3}
\]

These three vectors cancel each other:

\[
1 + \zeta_3 + \zeta_3^2 = 0
\]

Geometrically, this means that three arrows of equal length at angles \(0^\circ\), \(120^\circ\), and \(240^\circ\) balance each other perfectly.

\subsection{Near-Vanishing Sums}

Real audio data is noisy. Therefore, exact cancellation is unlikely.

Instead of searching only for:

\[
z_1 + z_2 + \dots + z_k = 0
\]

the prototype searches for:

\[
|z_1 + z_2 + \dots + z_k| < \varepsilon
\]

where \(\varepsilon\) is a small tolerance.

This means that the vectors do not need to sum to exactly zero. It is enough that their sum is close to zero.

\subsection{Interpretation}

A near-vanishing sum means that a group of phase components has a structured cancellation pattern.

In the context of underwater audio, this can indicate:

\begin{itemize}[leftmargin=*]
    \item structured periodic interference,
    \item repeated mechanical behavior,
    \item harmonic relationships,
    \item symmetric phase relationships.
\end{itemize}

The system marks these components as candidates for structured interference.

\subsection{Filtering and Reconstruction}

After the structured components are detected, the system can attenuate them.

This means it reduces their amplitude in the Fourier domain. Then the system applies the inverse Fourier transform to reconstruct the audio.

The result is a cleaned or residual version of the signal.

The goal is not to produce perfect separation. The goal is to make weak acoustic content easier to inspect.

\section{Final MVP Scope}

\subsection{Must-Have Features}

The final hackathon prototype should include the following core features:

\begin{enumerate}[leftmargin=*]
    \item load or upload an audio file,
    \item display the original waveform,
    \item compute the Fourier transform,
    \item display the frequency spectrum,
    \item receive detected phase-cancellation groups from the algorithm,
    \item visualize selected phase components on the unit circle,
    \item attenuate detected frequency components,
    \item reconstruct the cleaned audio,
    \item play the original audio,
    \item play the cleaned audio,
    \item show a simple explanation of the result.
\end{enumerate}

\subsection{Nice-to-Have Features}

The following features should only be added if the main demo is already stable:

\begin{itemize}[leftmargin=*]
    \item export cleaned audio,
    \item show confidence score,
    \item show multiple detected groups,
    \item animate vectors on the unit circle,
    \item compare original and cleaned frequency spectra,
    \item add a marine-themed interface,
    \item add a prepared sample gallery.
\end{itemize}

\subsection{Features to Avoid}

To save time, the team should not build:

\begin{itemize}[leftmargin=*]
    \item real-time streaming,
    \item a mobile app,
    \item user accounts,
    \item database storage,
    \item complex machine learning,
    \item species classification,
    \item scientific validation on a large dataset.
\end{itemize}

The goal is to build one clear, working, explainable prototype.

\section{Common Interface With the Algorithm}

\subsection{Why the Interface Matters}

Even though Vasilije is working on the algorithm independently, the rest of the team needs a clear interface. Without this, integration can become chaotic.

The team should agree on:

\begin{itemize}[leftmargin=*]
    \item what input the algorithm receives,
    \item what output it returns,
    \item how the backend calls the algorithm,
    \item what data the frontend needs for visualization.
\end{itemize}

\subsection{Suggested Algorithm Input}

The simplest input is a path to a WAV file:

\begin{verbatim}
input_audio.wav
\end{verbatim}

The backend should pass this path to the Julia script.

A simple command-line interface could look like:

\begin{verbatim}
julia phaseguard_engine.jl input_audio.wav output_analysis.json
\end{verbatim}

\subsection{Suggested Algorithm Output}

The algorithm should return a JSON file. The JSON should be simple and easy to parse.

Suggested structure:

\begin{verbatim}
{
  "sample_rate": 44100,
  "duration_seconds": 5.0,
  "dominant_components": [
    {
      "index": 12,
      "frequency": 120.0,
      "amplitude": 0.91,
      "phase": 0.04,
      "root_index": 0
    }
  ],
  "detected_groups": [
    {
      "group_id": 1,
      "component_indices": [12, 28, 44],
      "frequencies": [120.0, 240.0, 360.0],
      "phases": [0.04, 2.11, 4.19],
      "root_indices": [0, 21, 43],
      "residual": 0.08,
      "confidence": 0.91
    }
  ]
}
\end{verbatim}

\subsection{Minimum Required Output}

If time is very short, the minimum useful output is:

\begin{verbatim}
{
  "detected_groups": [
    {
      "frequencies": [120.0, 240.0, 360.0],
      "phases": [0.04, 2.11, 4.19],
      "residual": 0.08
    }
  ]
}
\end{verbatim}

This is enough for the backend to suppress frequencies and for the frontend to visualize the phase group.

\subsection{Fallback Output}

If live algorithm integration fails, the team should use a precomputed JSON file.

This is important for demo safety. The project should still be presentable even if one technical part fails during the live presentation.

\subsection{Calling the Julia Algorithm}

The simplest integration is through a command-line call:

\begin{verbatim}
subprocess.run([
    "julia",
    "phaseguard_engine.jl",
    input_path,
    output_json_path
])
\end{verbatim}

This is not the most elegant software architecture, but it is good for a hackathon because it is simple and practical.

Uros should make sure that:

\begin{itemize}[leftmargin=*]
    \item the Julia script path is configurable,
    \item errors are captured,
    \item timeout is used,
    \item invalid JSON does not crash the backend,
    \item a fallback JSON can be used if needed.
\end{itemize}

\subsection{Filtering and Reconstruction}

After the algorithm returns detected frequency components, Uros should perform attenuation and reconstruction.

The idea is:

\[
\text{Audio}
\rightarrow
\text{FFT}
\rightarrow
\text{Reduce selected components}
\rightarrow
\text{Inverse FFT}
\rightarrow
\text{Cleaned audio}
\]

The detected frequencies should not necessarily be removed completely. Complete removal can create artifacts. A safer first approach is attenuation.

For example:

\[
X_k^{cleaned} = 0.2 \cdot X_k
\]

for detected frequency bins.

This means the selected frequency components are reduced to 20\% of their original strength.

\subsection{Spectrum Data for Frontend}

The frontend needs spectrum data in a simple form.

Suggested structure:

\begin{verbatim}
[
  {
    "frequency": 120.0,
    "amplitude": 0.91,
    "detected": true
  },
  {
    "frequency": 240.0,
    "amplitude": 0.74,
    "detected": true
  }
]
\end{verbatim}

The frontend should not receive thousands of FFT bins. Uros should downsample or return only the most important points.

\subsection{Waveform Data for Frontend}

The waveform should also be downsampled.

Suggested structure:

\begin{verbatim}
[
  {
    "time": 0.00,
    "value": 0.01
  },
  {
    "time": 0.01,
    "value": -0.03
  }
]
\end{verbatim}

Returning too many points can make the frontend slow. A few thousand points are enough for visualization.

\subsection{Unit Circle Data}

The frontend needs data for plotting phases on the unit circle.

Suggested structure:

\begin{verbatim}
[
  {
    "x": 1.0,
    "y": 0.0,
    "phase": 0.0,
    "frequency": 120.0,
    "group_id": 1,
    "detected": true
  }
]
\end{verbatim}

This allows Rastko to draw points and vectors.

\subsection{Fallback Mode}

Uros should prepare fallback mode.

Fallback mode means:

\begin{itemize}[leftmargin=*]
    \item if live Julia call fails, load saved analysis JSON,
    \item if upload fails, use prepared sample audio,
    \item if reconstruction fails, still show analysis and plots.
\end{itemize}

This is very important. A hackathon demo should not depend on one fragile path.

\subsection{Error Handling}

The backend should return friendly errors such as:

\begin{verbatim}
{
  "status": "error",
  "message": "Could not process the audio file. Please use a WAV file."
}
\end{verbatim}

or:

\begin{verbatim}
{
  "status": "warning",
  "message": "No strong structured phase-cancellation group was detected."
}
\end{verbatim}

The frontend can display these messages instead of crashing.

\subsection{Uros Priority List}

Uros should focus on:

\begin{enumerate}[leftmargin=*]

    \item Julia script call,
